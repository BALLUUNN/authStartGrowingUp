# CONTRIBUTING

Документ описывает единые правила работы с кодовой базой, чтобы изменения были безопасными, воспроизводимыми и удобными для ревью.

## 1. Перед началом

- Проверьте актуальный `README.md` и `STYLE.md`.
- Для крупной задачи создайте issue с кратким дизайном решения.
- Для security-sensitive изменений заранее согласуйте подход с maintainers.

## 2. Ветки

- Создавайте feature-ветки от актуальной основной ветки.
- Рекомендованный формат имени:
  - `feature/<short-topic>`
  - `fix/<short-topic>`
  - `chore/<short-topic>`

## 3. Коммиты

- Делайте атомарные коммиты (одна логическая единица изменений).
- Пишите понятные сообщения коммитов.
- Рекомендуемый стиль:
  - `feat: add refresh token rotation`
  - `fix: prevent otp replay`
  - `chore: improve auth logs`

## 4. Обязательные проверки перед PR

- Код форматирован (`gofmt`).
- Пройдены тесты:
  - `go test ./...`
- Пройдены статические проверки:
  - `go vet ./...`
- Добавлены/обновлены тесты для измененной логики.
- Обновлена документация, если изменился API/поведение.

Для текущего bootstrap-сервиса базовая локальная проверка выглядит так:

```bash
go test ./...
go vet ./...
mkdir -p .tmp && go build -o .tmp/authStartGrowingUp ./cmd
go run ./cmd
```

## 5. Pull Request

В описании PR обязательно укажите:

- Что изменено.
- Почему это нужно.
- Риски и потенциальные side effects.
- Как проверялось (команды, сценарии, результаты).

Желательно добавить:
- чек-лист по безопасности (секреты, валидация, rate limit, логирование),
- миграционный план (если есть изменения схемы БД),
- rollback-подход для production.

## 6. Review policy

- Минимум один одобряющий review перед merge.
- Для изменений в auth/token/security логике: минимум два review (если доступно).
- Все discussion thread-ы должны быть закрыты.

## 7. Security disclosure

Если вы нашли уязвимость:

- Не открывайте публичный issue с деталями эксплуатации.
- Сообщите maintainers приватно.
- Подготовьте минимальный PoC и рекомендации по исправлению.

## 8. Что не принимается

- Коммиты с секретами/ключами.
- PR без тестов для критичной бизнес-логики.
- Смешивание несвязанных изменений в одном PR.
- Необоснованные breaking changes без миграционного плана.

## 9. Definition of Done

Изменение считается завершенным, если:

- пройдены тесты и проверки,
- покрыты edge cases,
- обновлена документация,
- PR прошел review и готов к безопасному деплою.
