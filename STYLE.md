# STYLE GUIDE

Руководство по стилю для auth-сервиса StartGrowingUp.

## 1. Принципы

- Security first: любой код рассматривается через призму безопасности.
- Явность важнее магии: предсказуемые имена и явные контракты.
- Малые модули: функции и пакеты с одной ответственностью.
- Production thinking: логируем, метрим, обрабатываем ошибки явно.

## 2. Стиль кода (Go)

- Следовать `gofmt` и стандартным Go-conventions.
- Имена:
  - пакеты: короткие и осмысленные (`auth`, `token`, `session`),
  - экспорты: PascalCase,
  - неэкспортируемые сущности: camelCase.
- Избегать сокращений, кроме общепринятых (`ctx`, `id`, `ttl`).
- Предпочитать ранний выход из функции для обработки ошибок.

## 3. Обработка ошибок

- Никаких silent-fail веток.
- Оборачивать ошибки с контекстом (`fmt.Errorf("...: %w", err)`).
- Не отдавать во внешний API внутренние детали (stack trace, SQL, секреты).
- Для API использовать стабильные machine-readable коды ошибок.

## 4. API и контракты

- Все публичные endpoint-ы versioned (`/v1/...`).
- Входные payload всегда валидируются.
- Ответы API:
  - однородный формат,
  - request/correlation id,
  - понятный код и сообщение ошибки.
- Идемпотентность для повторяемых операций (где применимо).

## 5. Безопасность

- Пароли хэшировать через Argon2id (или bcrypt с достаточным cost).
- OTP хранить и проверять с TTL, ограничением попыток, anti-replay.
- Access token короткоживущий, refresh token ротуемый.
- Секреты только из env/secret manager.
- Логи без персональных секретов и credential-данных.

## 6. Логирование и наблюдаемость

- Использовать структурированные JSON-логи.
- Каждое критичное действие в auth-сценариях логируется как security event.
- Обязательные поля логов: timestamp, level, request_id, actor_id (если есть), action, result.
- Метрики и трассировки должны покрывать ключевые auth-потоки.

## 7. Тесты

- Каждый багфикс сопровождается тестом, воспроизводящим регрессию.
- Unit-тесты для бизнес-логики обязательны.
- Integration-тесты для БД и токен/сессионных сценариев обязательны для критичных потоков.
- Тесты должны быть детерминированными.

## 8. Документация

- Изменения в публичных API отражать в README/доках в том же PR.
- Для нетривиальной логики добавлять короткие комментарии с объяснением почему.
- Обновлять runbook-и при изменениях в операционных процессах.

## 9. Pull Request hygiene

- Один PR = одна логическая задача.
- Короткий и ясный заголовок.
- Описание должно включать:
  - цель,
  - что изменено,
  - риски,
  - как проверялось.
